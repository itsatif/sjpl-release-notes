Dynamic Unit Cost Implementation

In a previous development phase, a feature was implemented to address the issue of providing users with the ability to adjust the unit cost according to specific sites. However, this feature was not released to the product due to clearance issues and pending approval. The feature includes an input field on the first step of the configuration page where users can add or update the unit cost. The unit cost is then utilized in the calculation of values such as consumption, savings, etc., on the dashboard through the init JouleTrack API.

The current unit cost is hard-coded as Rs 9 for all sites, except for Rs 6 for the LMW site. This limits the flexibility for users to customize and adjust the unit cost based on specific sites.

Impact

Users are unable to set site-specific unit costs, leading to inaccurate calculations and reporting on the Dashboard.

Expected Outcome

  1. Users should be able to input and update the unit cost for specific sites.
  2. The updated unit cost should be reflected in the calculations and values displayed on the dashboard.

Proposed Solution

  • Unit Cost Input Field: Implement an input field on the first step of the configuration page where users can add or update the unit cost for specific sites. Ensure that the input field accepts valid numerical values and provides appropriate validation feedback to users.
  • init JouleTrack API Integration: Integrate the unit cost parameter from the init JouleTrack API. Retrieve the user-defined unit cost for each site and pass it to the relevant calculations.
  • Dashboard Calculation Updates: Update the calculations and values displayed on the dashboard to incorporate the user-defined unit cost for each site. Ensure that the updated unit cost is utilized in the appropriate calculations for consumption, savings, and other relevant metrics on the Dashboard.

Invalid URL in "View Alerts" Button Dashboard

The "View Alerts" button in the dashboard displays an invalid URL. When users click on the "View Alerts" button, it leads to a broken or incorrect URL, preventing them from accessing the alerts page.

Impact

Users are unable to view alerts and take necessary actions, leading to potential oversight of critical information.

Proposed Solution

Fix the URL for the "View Alerts" button to correctly navigate users to the alerts page, enabling them to access and address alerts as intended.

Added Child-Based Routing

Currently, the configuration step 3 does not maintain its state upon reload when a specific component is selected. When users select a specific component in configuration step 3, the routing is enabled to maintain its state upon reload. This allows users to seamlessly continue the configuration process without losing their selected component.

Impact

Without child-based routing, users are forced to reselect the component every time they reload the configuration page, resulting in a suboptimal user experience and potential errors.

Proposed Solution

Implement child-based routing for configuration step 3 to retain the selected component's state upon page reload, ensuring a smoother configuration process for users.

Change Status Code of Obiviously API in JoulePulse Page

The Obiviously API on the JoulePulse page currently returns a status code of 403 in cases of recursive logout, causing confusion for users. To improve clarity and user experience, change the status code of the Obiviously API from 403 to 400 when users encounter the recursive logout issue on the JoulePulse page.

Impact

Users will receive a more accurate status code, helping them understand the reason for the issue and facilitating troubleshooting and error resolution.

Proposed Solution

Update the Obiviously API to return a status code of 400 when users experience the recursive logout issue on the JoulePulse page.
